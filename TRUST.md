## 0-d. 구매자 보호 · 안전거래 루프 (확정 2026-07-21)

> 게임 아이템 안전거래와 같이 **단계가 끝나야 다음 단계**. 보증·보험 문구 금지.  
> **PG 연동 전에는 실운영 안 함.** 입금 확인의 정답은 **PG 웹훅 → `paid`**.  
> **PG 없을 때 쓸 차선책 기능(운영 수동 입금 UX 등)을 앞으로 추가하지 않는다.**

### 흐름 (PG 기준)

1. **낙찰/성사** → `deal_status=awaiting_payment` · **입금 기한 기본 1시간** (`DEAL_PAYMENT_HOURS`) · **PG 결제 링크**
2. **PG 결제 성공(웹훅)** → `paid` → 판매자 **이전 가능** (입금 전 이전 금지)
3. 판매자 **이전 완료** `POST .../deal/mark-transferred` → `inspection` · **검수 기한 기본 48시간** (`DEAL_INSPECTION_HOURS`, 24–72)
4. 구매자 **인수하기** `POST .../deal/accept` → `completed` · **판매자 정산** (수수료 10%)
5. 검수 기한 내 **이의 없음** → 스케줄러 **자동 구매 확정·정산** (`process_deal_deadlines`)
6. 이의 `POST .../deal/dispute` → `disputed` (자동 확정 중지)
7. **1시간 미입금/결제 만료** → `payment_default` · 낙찰 무효 · 구매자 신용 감점

### 고지

- 입찰 UI · 매물 상세 안전거래 박스 · `GET /api/v1/config` → `payment_policy.deal`
- 카피: 「인수 확정 또는 기한 내 무이의 시 자동 확정·정산」 · 「입금은 PG 결제」

### 환경변수

| 키 | 기본 | 의미 |
|----|------|------|
| `DEAL_PAYMENT_HOURS` | 1 | 낙찰 후 입금 기한(시간) |
| `DEAL_INSPECTION_HOURS` | 48 | 이전 후 자동 확정 기한(시간) |

## 0. 거래 속도 · 회전율 (제품 포지션)

> **메이저 장기 경매가 아님.** 빠른 합의·빠른 입금·빠른 이전으로 **회전율**을 올린다.  
> (`PLATFORM.md` 「거래 속도 · 회전율」과 동일 확정)

### 왜 입금 기한을 짧게 잡나 (PG · 1시간)

1. **돈이 있으면 입찰한다** — 입찰·낙찰은 이미 결제 의사가 있다는 전제. “나중에 마련한다”는 우리 장이 아님.  
2. **장난 vs 진심** — 짧은 기한 안에 결제가 안 되면 의사(또는 자금) 없음으로 본다.  
3. **차순위 보호·회전** — 미결제 시 다음 순위. 1순위에 매물을 묶어 두지 않는다.  
4. **메이저 옥션이 아님** — 24·48시간 유예는 채택하지 않는다.  
5. **실운영은 PG 연동 후** — 결제 링크·웹훅·1시간 만료. **PG 없는 수동 입금 차선책은 두지 않음.**

| 항목 | 기본값 | 비고 |
|------|--------|------|
| 낙찰 후 결제 기한 | **1시간** (`DEAL_PAYMENT_HOURS`) | PG 결제 링크 |
| 미결제 시 | 낙찰 무효 · 신용 반영 · (PG 후) 차순위 자동화 목표 | 운영 수동 입금 경로 없음 |
| 고지 의무 | 입찰·매물 상세 | 안전거래 단계 사전 고지 |
| 기본 경매 기간 | **3~7일** | 장기간 방치 UX 지양 |
| 입찰 의미 | 결제 의사 + 기한 준수 | 허위·장난 입찰 시 이용 제한 |
| 자산 이전 | **PG 결제 확인 후** | 판매자 보호 1순위 |

카피 톤 예:  
> “이기면 **1시간 안에 PG로 결제**해야 합니다. 미결제 시 낙찰 무효. 결제 확인 후 이전·검수·인수(또는 48시간 자동 확정).”

**구현:**  
- 자동 낙찰 · deal 상태 머신 · 스케줄러 기한 처리 있음  
- **결제 링크·웹훅→`paid`** = PG 연동 시 연결 (`mark_deal_paid`는 웹훅 전용, 관리자 수동 입금 API 없음)

## 0-b. 사이트 내 신용 점수 (자동 · 공개 고지)

> **Lv0~Lv3 = 자격**, **사이트 내 신용 점수 = 거래 행동 평판**. 둘은 별개. UI 표기는 **Lv** (예: Lv2).  
> 점수는 **0~100**, 이벤트 발생 시 **자동 재계산**. 산식은 회원에게 공개 (`/guide/credit.html`, `GET /api/v1/credit-policy`).

### 산식 (기본 30점 · 거래 중심)

> 신원만 채우면 높아 보이지 않게. 사기·평판 판별은 **성사·미입금 이력** 비중을 키움.

| 항목 | 점수 | 비고 |
|------|------|------|
| 기본 | **+30** | 가입 시 (낮게 시작) |
| Lv2 신원 (실명·휴대폰) | **+5** | |
| Lv3 정산 계좌 | **+5** | 가입+Lv2+Lv3 ≈ **40** → 보통/신규 |
| 판매 성사 1건 | **+12** | 최대 +36 (3건) |
| 구매 성사 1건 | **+12** | 최대 +36 |
| 정시 입금 (안내 기한 준수 · PG 후 1시간 목표) | **+8** | 최대 +24 |
| 낙찰 후 미입금 | **−40** | 건당 · 하한 0 |

최종 = clamp(합계, 0, 100)

### 등급

| 점수 | 등급 |
|------|------|
| 90–100 | 최고 |
| 75–89 | 우수 |
| 60–74 | 신뢰 |
| 40–59 | 보통 |
| 20–39 | 신규 |
| 0–19 | 주의 |

### 자동 반영 시점

- 프로필 Lv2 / 정산 Lv3 저장 → 재계산  
- 성사(`finalize_sale`) → 판매자 sold+1, 구매자 bought+1 · on_time+1 (MVP: 성사 시 정시 가점)  
- 운영 미입금 기록 `POST /admin/credit/default` → defaults+1  

### 공개 범위

- 타인: `score` · `label` · 성사/미입금 **건수** (세부 breakdown 비공개)  
- 본인: breakdown 포함 (`GET /me`)  
- 매물 상세 판매자 카드에 배지 표시

## 1. 신뢰 레벨

| 레벨 | 이름 | 조건 | 할 수 있는 것 |
|------|------|------|----------------|
| **Lv0** | 가입 | 이메일 + 비밀번호 | 공개 매물 열람, 관심 등록(경량) |
| **Lv1** | 이메일 인증 | Lv0 + 이메일 코드 확인 | **입찰(가격 쓰기)** · 프로필 입력 |
| **Lv2** | 신원 프로필 | Lv1 + **실명** + **휴대폰** + 역할 | **매물 등록**, 낙찰 후 **결제·인수**, 신고 |
| **Lv3** | 거래 준비 | Lv2 + **정산 계좌**(예금주·계좌 식별) | **판매자 정산·성사 확정** (PG 후 수령) |

> **마찰 최소화 (2026-07-21):** 입찰 허들은 Lv1까지. 실명·폰(Lv2)은 **낙찰 후**, 계좌(Lv3)는 **판매자 정산** 때.  
> `can_bid` = email · `can_fulfill_purchase` = Lv2 · `can_list` = Lv2+판매자공개 · `can_close_deal` = Lv3.

레벨은 API `user.trust.level` / `user.trust.can_*` 로 클라이언트가 읽는다.

## 2. 단계별 수집 항목

### 관심·사전등록 (랜딩 buy — L0 가능)
- 이메일, 관심 카테고리, 예산(선택)
- 목적: 매칭 연락. **고액 입찰 권한은 없음**

### 계정 가입 (L0 → L1)
- 이메일, 비밀번호, 표시 이름(선택) **또는 SNS** (카카오·구글·깃허브)
- SNS는 **가입·로그인 간소화만** — 계좌인증만으로 대체하지 **않음**
- **생년월일(필수)** + 「만 14세 이상」 확인 — SNS 직후에도 없으면 별도 화면 (`#age` / `PUT /me/birth-date`)
- **만 14세 미만은 가입 거절** (법정대리인 동의 경로 없음)
- **이메일 6자리 코드 인증** 필수 후 L1 (SNS가 검증 이메일을 주면 생략 가능)

### 판매자 공개 신원 (필수 · 통신판매중개 정보 제공 · 2026-07-19)

> 구매자가 **판매자가 누구인지** 확인할 수 있어야 함.  
> 없으면 중개 플랫폼이 판매자로 간주될 위험 → **매물 올리기 전 필수**.

| 항목 | 내용 |
|------|------|
| API | `PUT /api/v1/me/seller-identity` |
| 게이트 | `trust.can_list` = Lv2 + **seller_identity_complete** |
| 개인 | 공개 성명 · 연락 이메일·전화 · (권장) 주소 · 통신판매 신고번호(있으면) |
| 사업자 | 상호 · 대표 · 사업자번호 · 주소 · 연락처 · 신고번호(있으면) |
| 공개 위치 | 매물 상세 `project.seller.identity` · UI 판매자 정보 카드 |
| 미신고 | 「통신판매업 신고번호 미등록」으로 표시 |
| **자기 신고** | 판매자 신원은 **이용자 입력** (PASS/사업자 진위 자동검증 전). UI에 「판매자 신고 정보」 취지 고지 |
| **연락처 마스킹** | 공개: 전화 `010-1234-****` · 이메일 `ab***@domain.com` |
| **전체 연락처** | **낙찰(성사) 구매자** + **판매자 본인**만 (`contact_revealed: true`) |

#### 개인정보 원칙 (확정 · 위반 금지)

> **전체 전화·이메일을 불특정 다수에게 공개하는 것은 개인정보 유출에 해당할 수 있다. 하지 않는다.**

| 규칙 | 내용 |
|------|------|
| 금지 | 비로그인·비낙찰 이용자에게 판매자 **전체 연락처** 노출 |
| 허용(신원) | 상호/성명, 사업자번호, 통신판매 신고번호 등 **거래 상대 특정**에 필요한 최소 정보 |
| 연락처 | 공개 = **마스킹만** · 전체 = **성사 당사자(낙찰 구매자·판매자)** 한정 |
| API | `public_seller_identity(..., reveal_contact=False)` 기본 · True는 권한 검사 후만 |
| Lv2 실명·휴대폰 | 운영·게이트용 · **공개 피드에 그대로 노출 금지** |

|------|------|

### 신원 프로필 (L1 → L2) — 매물 등록 전 필수
| 필드 | 필수 | 용도 |
|------|------|------|

| 역할 (role) | ✅ | seller / buyer / both |
| 표시 이름 | 선택 | UI 표기 (실명과 분리 가능) |

### 거래 준비 (L2 → L3) — 성사 직전
| 필드 | 필수 | 용도 |
|------|------|------|

| 사업자 여부 | 선택 | 세금계산서 등 |

> SMS 본인인증·PASS·에스크로 PG는 **다음 단계**. 지금은 수집 + 게이트 + 운영 수동 검수.

## 5. 이메일 인증 (MVP)

- 가입·재발송 시 **6자리 코드**, 유효 **30분**
- 로컬 개발: `EMAIL_DEV_MODE=1` 이면 응답에 `dev_email_code` 포함 가능
- **운영 필수:** `EMAIL_DEV_MODE=0` + `EMAIL_CODE_FALLBACK=0` (SMTP 실패 시에도 코드 API 비노출) + 실 메일(Resend/SMTP)
- 아이디 찾기: 마스킹 힌트만 · IP 요청 제한 · 원문 이메일 미반환
