## 0. 거래 속도 · 회전율 (제품 포지션)

> **메이저 장기 경매가 아님.** 빠른 합의·빠른 입금·빠른 이전으로 **회전율**을 올린다.  
> (`PLATFORM.md` 「거래 속도 · 회전율」과 동일 확정)

### 왜 입금 기한을 짧게 잡나 (PG 후 목표: 1시간)

1. **돈이 있으면 입찰한다** — 입찰·낙찰은 이미 결제 의사가 있다는 전제. “나중에 마련한다”는 우리 장이 아님.  
2. **장난 vs 진심** — 짧은 기한 안에 입금이 안 되면 의사(또는 자금) 없음으로 본다.  
3. **차순위 보호·회전** — 미입금 시 다음 순위 연락. 1순위에 매물을 묶어 두지 않는다.  
4. **메이저 옥션이 아님** — 24·48시간 유예는 채택하지 않는다.  
5. **지금(pre-PG)** — 자동 1시간 타이머 없음. 유저 대면 문구는 “안내 따라 신속 입금” + “PG 후 1시간 목표”.

| 항목 | 기본값 | 비고 |
|------|--------|------|
| 낙찰 후 결제 기한 | **안내에 따른 신속 입금** | **PG 후 목표: 1시간**. 지금은 자동 타이머 없음 |
| 미결제 시 | 낙찰 무효 · 차순위 가능 | **PG 전:** 운영 수동. **PG 후:** 2순위 자동 목표 |
| 고지 의무 | 입찰 화면·약관 | 현재 단계(pre-PG)와 목표를 **분리 고지** (구현 없는 자동화 약속 금지) |
| 기본 경매 기간 | **3~7일** | 등록 시 기본값; 장기간 방치 UX 지양 |
| 입찰 의미 | 결제 의사 + 안내 기한 준수 + 차순위 전환 동의 | 허위·장난 입찰 시 이용 제한 |
| 자산 이전 | 입금(또는 에스크로) **확인 후** | 판매자 보호 1순위 |

카피 톤 예 (입찰 폼 · pre-PG):  
> “이기면 **안내에 따라 빠르게 입금**해야 합니다. 미입금 시 낙찰 무효·다음 순위 전환 가능. **1시간 자동 타이머·결제 링크는 PG 연동 후.** 결제 준비된 분만 입찰해 주세요.”

**구현 상태 (2026-07-19 갱신):**  
- 마감 시각 → **자동 낙찰** 있음 (`process_expired_auctions`)  
- **결제 링크 · 1시간 타이머 · 2순위 자동** = **PG 연동 후** 백로그 (`docs/나중_할일_BACKLOG.md` §B)  
- **유저 대면 카피·약관 v2.1** 은 위 구현 상태와 맞춤 (자동화 미구현을 사실처럼 쓰지 않음)

## 0-b. 사이트 내 신용 점수 (자동 · 공개 고지)

> **Lv0~Lv3 = 자격**, **사이트 내 신용 점수 = 거래 행동 평판**. 둘은 별개. UI 표기는 **Lv** (예: Lv2).  
> 점수는 **0~100**, 이벤트 발생 시 **자동 재계산**. 산식은 회원에게 공개 (`/guide/credit.html`, `GET /api/v1/credit-policy`).

### 산식 (기본 30점 · 거래 중심)

> 신원만 채우면 높아 보이지 않게. 사기·평판 판별은 **성사·미입금 이력** 비중을 키움.

| 항목 | 점수 | 비고 |
|------|------|------|
| 기본 | **+30** | 가입 시 (낮게 시작) |
| Lv2 신원 (실명·휴대폰) | **+5** | |
| Lv3 정산 계좌 | **+5** | 가입+Lv2+Lv3 ≈ **40** → 보통/신규 |
| 판매 성사 1건 | **+12** | 최대 +36 (3건) |
| 구매 성사 1건 | **+12** | 최대 +36 |
| 정시 입금 (안내 기한 준수 · PG 후 1시간 목표) | **+8** | 최대 +24 |
| 낙찰 후 미입금 | **−40** | 건당 · 하한 0 |

최종 = clamp(합계, 0, 100)

### 등급

| 점수 | 등급 |
|------|------|
| 90–100 | 최고 |
| 75–89 | 우수 |
| 60–74 | 신뢰 |
| 40–59 | 보통 |
| 20–39 | 신규 |
| 0–19 | 주의 |

### 자동 반영 시점

- 프로필 Lv2 / 정산 Lv3 저장 → 재계산  
- 성사(`finalize_sale`) → 판매자 sold+1, 구매자 bought+1 · on_time+1 (MVP: 성사 시 정시 가점)  
- 운영 미입금 기록 `POST /admin/credit/default` → defaults+1  

### 공개 범위

- 타인: `score` · `label` · 성사/미입금 **건수** (세부 breakdown 비공개)  
- 본인: breakdown 포함 (`GET /me`)  
- 매물 상세 판매자 카드에 배지 표시

## 1. 신뢰 레벨

| 레벨 | 이름 | 조건 | 할 수 있는 것 |
|------|------|------|----------------|
| **Lv0** | 가입 | 이메일 + 비밀번호 | 공개 매물 열람, 관심 등록(경량) |

## 2. 단계별 수집 항목

### 관심·사전등록 (랜딩 buy — L0 가능)
- 이메일, 관심 카테고리, 예산(선택)
- 목적: 매칭 연락. **고액 입찰 권한은 없음**

### 계정 가입 (L0 → L1)
- 이메일, 비밀번호, 표시 이름(선택) **또는 SNS** (카카오·구글·깃허브)
- SNS는 **가입·로그인 간소화만** — 계좌인증만으로 대체하지 **않음**
- **생년월일(필수)** + 「만 14세 이상」 확인 — SNS 직후에도 없으면 별도 화면 (`#age` / `PUT /me/birth-date`)
- **만 14세 미만은 가입 거절** (법정대리인 동의 경로 없음)
- **이메일 6자리 코드 인증** 필수 후 L1 (SNS가 검증 이메일을 주면 생략 가능)

### 판매자 공개 신원 (필수 · 통신판매중개 정보 제공 · 2026-07-19)

> 구매자가 **판매자가 누구인지** 확인할 수 있어야 함.  
> 없으면 중개 플랫폼이 판매자로 간주될 위험 → **매물 올리기 전 필수**.

| 항목 | 내용 |
|------|------|
| API | `PUT /api/v1/me/seller-identity` |
| 게이트 | `trust.can_list` = Lv2 + **seller_identity_complete** |
| 개인 | 공개 성명 · 연락 이메일·전화 · (권장) 주소 · 통신판매 신고번호(있으면) |
| 사업자 | 상호 · 대표 · 사업자번호 · 주소 · 연락처 · 신고번호(있으면) |
| 공개 위치 | 매물 상세 `project.seller.identity` · UI 판매자 정보 카드 |
| 미신고 | 「통신판매업 신고번호 미등록」으로 표시 |
| **자기 신고** | 판매자 신원은 **이용자 입력** (PASS/사업자 진위 자동검증 전). UI에 「판매자 신고 정보」 취지 고지 |
| **연락처 마스킹** | 공개: 전화 `010-1234-****` · 이메일 `ab***@domain.com` |
| **전체 연락처** | **낙찰(성사) 구매자** + **판매자 본인**만 (`contact_revealed: true`) |

#### 개인정보 원칙 (확정 · 위반 금지)

> **전체 전화·이메일을 불특정 다수에게 공개하는 것은 개인정보 유출에 해당할 수 있다. 하지 않는다.**

| 규칙 | 내용 |
|------|------|
| 금지 | 비로그인·비낙찰 이용자에게 판매자 **전체 연락처** 노출 |
| 허용(신원) | 상호/성명, 사업자번호, 통신판매 신고번호 등 **거래 상대 특정**에 필요한 최소 정보 |
| 연락처 | 공개 = **마스킹만** · 전체 = **성사 당사자(낙찰 구매자·판매자)** 한정 |
| API | `public_seller_identity(..., reveal_contact=False)` 기본 · True는 권한 검사 후만 |
| Lv2 실명·휴대폰 | 운영·게이트용 · **공개 피드에 그대로 노출 금지** |

### 매물 등록 최소 가이드 (필수 · 2026-07-19)

등록 API·앱 폼에서 **전부 충족해야 접수**됩니다. (서버 강제)

| 항목 | 내용 |
|------|------|
| 작동 확인 | 「최소한 돌아가는 코드·데모인지 직접 확인」 체크 |
| 라이선스 | 텍스트 필수 (MIT/GPL 등 또는 사유·양도 조건) + 기재 확인 체크 |
| 권한 | 「팔 권한이 있는 자산」 체크 (도용 아님) |
| 저장 | `license_note` · `seller_attest_json` |
| 의미 | **판매자 서약** — 플랫폼 보증 아님. 허위 체크 시 제재·신고 대상 |

### 신원 프로필 (L1 → L2) — 매물 등록 전 필수
| 필드 | 필수 | 용도 |
|------|------|------|
| 실명 (real_name) | ✅ | 거래 당사자 식별 · 분쟁 |
| 휴대폰 (phone) | ✅ | 연락·사기 억제 (형식 검증) |
| 역할 (role) | ✅ | seller / buyer / both |
| 표시 이름 | 선택 | UI 표기 (실명과 분리 가능) |

### 거래 준비 (L2 → L3) — 성사 직전
| 필드 | 필수 | 용도 |
|------|------|------|
| 예금주 | ✅ | 수수료·정산 |
| 은행명 | ✅ | 정산 |
| 계좌 식별 | ✅ | 정산. **저장 시 암호화** (`encrypt_settlement_account`, APP_SECRET/JWT_SECRET 파생). API는 마스킹만 반환 |
| 사업자 여부 | 선택 | 세금계산서 등 |

> SMS 본인인증·PASS·에스크로 PG는 **다음 단계**. 지금은 수집 + 게이트 + 운영 수동 검수.

## 5. 이메일 인증 (MVP)

- 가입·재발송 시 **6자리 코드**, 유효 **30분**
- 실 SMTP 전: `EMAIL_DEV_MODE=1`(기본 개발)이면 응답에 `dev_email_code` 포함 — **운영에서는 반드시 끄고 메일 발송 연동**
- 운영: `EMAIL_DEV_MODE=0` + 향후 메일 프로바이더
